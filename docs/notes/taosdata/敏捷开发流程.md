# 敏捷开发流程

- 敏捷开发流程：`需求 -> 飞书工作项`

1. 加上 `FS Required` 标签的飞书工作项必须准备用户行为和多泳道时序图的设计文档`功能设计文档 FS`，放入 `总设计文档`下面的相关模块文档下面，并和技术小组长和组长确认飞书工作项的`人天`。
2. 完成`功能设计文档FS`后需要进行专门方案审核会议，至少包括技术小组长和组长，如果需要平台组配合工作需要包括平台组相关人员。
3. 提交代码，`PR` 上面加上 `JIRA（飞书项目）`链接。
4. 解决所有 `AI` 的建议后按照要求`准备审核资料`并在飞书项目发起提交 PR 审核，。如果 PR 不过不是自己的功能引起的，请找技术小组长推动解决。

   - 审核资料在`飞书工作项-开发过程中填写`，包括以下内容：
     - `功能设计文档FS`的文档链接
     - github 代码仓库上的`PR`链接
     - github 文档仓库上的`PR`链接
     - 发版相关的`测试文档链接`，含测试结果

5. 质量小组长审核 JIRA（飞书项目）中的`测试用例`。
6. 技术小组长审核具体代码和文档，可以发起`代码审查会议`，如果有必须包括技术小组长和组长。
7. 组长审核并`合并`到 dev 或者 main 分支。

8. 代码冻结日（Code Frozen）

   - 每个开发迭代周期结束前的三天，比如周五发版，周三下班前 18 点代码冻结，合并到 dev 分支。
   - 代码冻结是改进和特性截止提交的日期，这天以后不能再提交大的功能代码，之后这个开发周期的任务只能提交缺陷解决。
   - 除了特别审批的改进和特性以外都不能再提交相关代码，特别严重缺陷除外。

9. 迭代评审总结会
   - 评审环境在 `staging` 环境（预生产环境）。
   - 改进和特性的评审列表早晨 9 点发布，之后每个改进和特性都需要 owner 在评审环境准备。
   - 在代码冻结日后的一天下午 15 点 - 16 点 30 召开。
   - 会议内容是评审迭代已经提交的功能，包括迭代已经提交功能的演示，评定这个迭代已经提交功能是否满足“完成的定义”。如果不满足“完成的定义”，从 UI 入口撤掉该改进和特性。
   - 每次迭代总结会完成后列出待办事项。
